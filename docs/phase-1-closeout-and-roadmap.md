## 立即建议

1. 先推送到自己的私有远程仓库，保存阶段成果和标签：

   ```bash
   git push origin master --tags
   ```

2. 在正式开源前，不要直接把当前本地配置文件原样提交。应先拆出 `appsettings.example.json` 或 README 配置说明。

3. 保留 `v0.1.0-alpha.1` 作为第一阶段里程碑，后续所有清理、文档、开源准备都从这个点继续推进。

## 开源前必须完成

- 许可证选择：建议优先考虑 `Apache-2.0`；如果特别担心别人拿去闭源 SaaS 化，可以再评估 `AGPL-3.0`。
- 配置脱敏：移除仓库中真实连接串、JWT 密钥、Redis 地址、本机路径，提供 example 配置。
- README 完整化：补上项目定位、功能截图、快速启动、默认账号、架构图、MCP/客户端说明。
- 安装体验验证：从干净目录 clone 后，按 README 能完成数据库初始化、后端启动、前端启动、登录。
- 生成项目验证：用内置示例项目完整跑一次初始化、全量生成、增量生成、前后端启动。
- 开源物料：补 `CHANGELOG.md`、`CONTRIBUTING.md`、Issue 模板、Roadmap。
- 截图和演示：至少准备登录页、项目管理、实体设计器、字段配置、生成结果、客户端启动能力的截图。
- 历史检查：确认 git 历史中没有提交过敏感信息；如果有，需要在公开前清理历史。

## 第二阶段优先级

### P0：发布卫生和可复现环境

- 建立一键冒烟脚本：构建后端、构建前端、运行 Migrator、启动 WebApi、启动 Vue、验证核心接口。
- 增加 example 配置：WebApi、Migrator、MCP Server、Tauri/LocalAgent 都要有可复制模板。
- 明确本地数据库策略：Sqlite 默认路径、MySQL/SqlServer 连接串示例、初始化失败排查。
- 建立发布清单：每次发布前固定跑同一套检查，避免靠记忆。

### P1：代码生成核心稳定性

- 继续强化字段排序：查询表单、列表、添加编辑页、详情页、子表字段都要稳定保持排序。
- 完善子表增量更新：主表增量生成时，能正确带动子表字段变更。
- 模板版本管理：模板变更要能追踪版本，并提供当前数据库模板更新脚本。
- 控件属性体系：把每种控件的字段用途、生成影响、前端表现整理成规范，给 MCP 和 UI 共用。
- 跨数据库验证：Sqlite、MySQL、SqlServer 至少各跑一套基础生成流程。

### P2：MCP 和 AI 工作流

- MCP 工具能力补齐：项目、模块、实体、字段、子表、初始化、启动、全量生成、增量生成。
- 结构化错误返回：MCP 调用失败要能区分参数错误、业务错误、生成错误、启动错误。
- 项目结构感知：MCP 能实时返回某个生成项目的实体、字段、关联、菜单和生成状态。
- 增加 MCP 冒烟测试：用自然语言流程对应的工具调用自动跑一遍，不让智能体直接改生成项目元数据。
- Skill 文档同步：MCP 工具参数、字段控件属性、使用边界要写进 Codex skill。

### P3：客户端与 LocalAgent

- 统一核心逻辑：Web 端和客户端调用同一套服务端 API、同一套后端业务逻辑，客户端只负责本地能力桥接。
- 本地能力清单：打开 cmd、启动前端、启动后端、本地路径选择、Sqlite 路径处理、进程状态查询。
- 进程生命周期：启动、停止、重启、已启动检测、日志查看、端口占用提示。
- 服务端部署场景：服务端前端可以部署到服务器，客户端 WebView 加载远程地址时仍可调用本地 sidecar 能力。
- 安全边界：本地能力只允许可信前端调用，避免任意网页调用 sidecar。

### P4：产品体验

- 标签页：关闭、缓存、不缓存页、详情页和编辑页行为保持清晰。
- 主题切换：暗色、浅色、跟随系统，确保 Element Plus 组件统一。
- 列表生命周期：从新增、编辑、详情返回列表时自动刷新，避免路由 push 造成状态混乱。
- 登录页和首页：保持现在更强的品牌感，同时不要影响首屏性能。
- 字典、菜单、权限、角色、租户等系统页面继续做一致性打磨。

### P5：测试体系

- 后端：补核心 Application 服务测试，尤其是模板生成、增量生成、项目初始化。
- 前端：补最小 E2E 流程，覆盖登录、项目列表、实体列表、字段编辑、生成代码。
- 生成项目：维护一个小型样例项目，作为每次发布的生成结果验收对象。
- 客户端：验证 WebView 加载本地前端和远程前端两种模式。

### P6：社区和商业化准备

- 先开源核心框架和基础生成能力，保持项目可信、可跑、可学。
- 后续可以把高级模板、企业级权限、多租户增强、可视化设计器增强、商业支持作为商业化方向。
- 做一份对比说明：CodeMaster 与若依、SmartCoder、ABP、低代码平台的差异。
- 准备首篇发布文章：讲清楚“为什么做 CodeMaster、它解决什么问题、为什么需要 MCP + LocalAgent”。

## 推荐执行顺序

1. 私有远程推送和标签备份。
2. 配置脱敏与 example 配置补齐。
3. README、CHANGELOG、Roadmap、截图补齐。
4. 干净环境从零启动验证。
5. MCP 工具和 Skill 再跑一轮自然语言冒烟。
6. 客户端 LocalAgent 远程前端场景验证。
7. 选择许可证并公开仓库。

## 开源 alpha 完成标准

- 新用户 clone 后，按 README 可以在 30 分钟内完成启动。
- 没有真实密钥、真实数据库密码、本机私有路径出现在仓库中。
- `dotnet build CodeMaster.sln`、`npm run build`、Migrator 初始化都能稳定通过。
- 至少一个内置示例项目能完成初始化、生成、启动和登录。
- README 能让别人明白 CodeMaster 的定位：不是若依换皮，而是面向 AI/MCP/本地客户端能力的新一代快速开发框架。
